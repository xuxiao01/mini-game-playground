# 小游戏原型文档生成 Skill

## 目标

你需要根据当前项目中某个小游戏的真实代码，生成一份适合团队内部复用和其他项目参考的「小游戏玩法文档」。

这份文档不是简单的 README，也不是纯技术说明，而是用于沉淀小游戏原型设计，包括玩法规则、交互流程、音效使用、动效设计、视觉样式、数据结构和可复用点。

请优先基于真实代码分析，不要凭空编造不存在的功能。如果代码中无法确认的信息，请标记为「待补充」。

---

## 1. 分析范围

请优先分析当前小游戏相关文件，包括但不限于：

- 页面入口文件
- 游戏主组件
- 子组件
- 静态数据 / 题库数据
- 类型定义
- 样式文件
- 音效资源
- 动画相关代码
- 工具函数
- 配置文件

如果项目中存在多个小游戏，请先识别当前目标小游戏，并只分析该小游戏相关代码。

游戏 id 以 `src/games/registry.ts` 中 `gameRegistry[].id` 为准；代码目录通常为 `src/games/{id}/`（若 id 与目录名不一致，文档中需注明映射关系）。

## 一、游戏定位

说明这个小游戏主要用于什么场景。

需要包括：

- 游戏类型，例如记忆类、识字类、匹配类、听音辨字类、反应类等
- 面向用户，例如儿童、低龄学习用户、内部试玩人员等
- 核心训练目标，例如识字、记忆力、观察力、词语搭配、听辨能力等
- 适合放在哪类产品中，例如儿童教育小程序、识字 App、H5 活动页等

示例：

> 本游戏是一个记忆匹配类小游戏，用户需要翻开卡牌并找到相同文字的卡片组合。游戏主要训练儿童的文字识别能力、短时记忆能力和观察能力，适合用于识字类小程序或儿童教育产品中的趣味练习模块。

## 二、核心玩法规则

这是最重要的部分，请写清楚用户到底怎么玩。

需要包括：

- 游戏开始时页面是什么状态
- 用户每一步可以做什么
- 点击 / 拖拽 / 选择后会发生什么
- 成功条件是什么
- 失败或不匹配时怎么处理
- 游戏什么时候结束
- 是否有分数、奖励、关卡、倒计时、步数限制
- 是否支持重新开始

请尽量用自然语言解释，不要只贴代码。

如果有流程，可以写成：

1. 页面初始化后，系统生成若干张卡牌。
2. 所有卡牌默认背面朝上。
3. 用户点击第一张卡牌后，卡牌翻转并展示文字。
4. 用户点击第二张卡牌后，系统判断两张卡牌文字是否相同。
5. 如果相同，两张卡牌保持正面展示，并计入得分。
6. 如果不同，两张卡牌短暂停留后翻回背面。
7. 当所有卡牌都匹配成功后，游戏结束并展示得分。

## 四、交互状态说明

请列出游戏中主要状态。

可以使用表格：

| 状态     | 触发时机         | 页面表现                     | 后续行为       |
| -------- | ---------------- | ---------------------------- | -------------- |
| 初始状态 | 页面加载完成     | 卡牌全部背面展示             | 等待用户点击   |
| 翻开状态 | 用户点击卡牌     | 卡牌翻转展示文字             | 等待第二张卡牌 |

| 游戏完成 | 所有卡牌匹配成功 | 展示结算区域                 | 可重新开始     |

如果代码里状态命名明确，请尽量对应代码里的变量名。

输出表格：

| ------------ | ------------------------------ | -------------------- | ---------------- | -------------------- |

## 七、动效设计说明

请分析游戏中使用到的动画和动效。

包括：

- 是否使用 GSAP / GASP 动画库
- 是否使用 CSS transition / animation
- 是否使用小程序 animation API
- 是否有翻转、淡入、缩放、抖动、弹跳、洗牌、飞入、闪光等效果
- 动效的触发时机和持续时间

输出表格：

| 动效名称     | 触发时机         | 实现方式                | 页面表现               | 备注             |
| ------------ | ---------------- | ----------------------- | ---------------------- | ---------------- |
| 卡牌翻转     | 点击卡牌         | CSS 3D transform / GSAP | 卡牌从背面翻到正面     | 核心动效         |
| 匹配成功反馈 | 两张卡牌匹配成功 | scale / glow / bounce   | 卡牌轻微放大或发光     | 增强奖励感       |
| 匹配失败反馈 | 两张卡牌不匹配   | shake / flip back       | 卡牌轻微抖动后翻回     | 错误反馈不宜过重 |
| 洗牌动效     | 游戏开始或重开   | shuffle / stagger       | 卡牌错位移动后重新排列 | 增强开局仪式感   |
| 结算出现     | 游戏完成         | fade in / slide up      | 结算区域淡入出现       | 提升完成感       |

如果代码中没有明确动效，请写「当前代码中未发现明确动效实现」，并补充合理的动效建议。

## 八、视觉样式说明

请总结当前小游戏的视觉风格。

需要包括：

- 主色调
- 背景样式
- 卡片样式
- 按钮样式
- 圆角、阴影、边框
- 字体大小和层级
- 是否偏儿童风、科技风、卡通风、后台工具风
- 是否适合移动端 / H5 / 小程序

输出可以包括：

### 当前视觉特征

- 页面整体采用浅色背景。
- 游戏区域使用白色或浅蓝色容器。
- 卡牌使用圆角卡片样式。
- 选中或成功状态使用高亮色。
- 按钮使用主色填充，强调可点击性。

### 可优化建议

- 儿童教育场景可增加更柔和的色彩和卡通反馈。
- 卡牌背面可以增加统一图案，提升游戏感。
- 成功匹配可以增加星星、徽章、闪光等正反馈元素。
- 错误反馈应尽量轻柔，避免强烈红色或刺耳动效。
